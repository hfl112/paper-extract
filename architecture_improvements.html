Workspace Directory Architecture

The data files and audit logs reside strictly under a unified, sandboxed layout:

data/collections/<collection_name>/ # Active collection root
├── collection.json # Collection manifest containing articles registry
├── articles.csv # Tabular flat index of identifiers
├── articles/ # Structured repositories
│ └── <article_id>/ # Single paper container (e.g. doi_10_1002_pbc_21368)
│ ├── article.json # Single source of truth (metadata, sections, extractions)
│ └── article.pdf # Optional local paper PDF
└── logs/ # Immutable audits trail
├── search_<ts>.json # Search parameters and additions
├── fetch_<ts>.json # Fetch execution summaries
└── status_<ts>.json # Periodic health reporting audits

Glossary Definitions

Consistent terminology blocks LLMs from making incorrect assumptions (e.g., trying to write to external services).

  • Collection: A folder on disk containing manifests and articles. Never referred to as a "dataset" or "database".
  • Article JSON: The single source of truth for an article's data (metadata, XML sections, extractions).
  • Audit Log: Immutable records of execution inputs/outputs in logs/ directory.

Anti-Patterns Checked

We set strict boundaries to avoid standard AI agent failures:

  • No credentials auto-fill: Agents are blocked from attempting to log in, scrape paywalled portals natively, or store keys.
  • Structured over raw PDF: Agent must rely on the structured sections dictionary in article.json first, avoiding expensive PDF parsing / OCR.
